# MindHelper: шпаргалка для дипломной работы

Тема ВКР: «Разработка вопросно-ответной системы для предварительной психологической диагностики на основе нейросети».

Файл нужен как внутренняя памятка для написания диплома. Его не обязательно вставлять в работу целиком, но из него удобно брать формулировки для глав 2-4.

## 1. Актуальная идея проекта

MindHelper — это полноформатный сервис первичной психологической поддержки, который объединяет:

- веб-интерфейс с регистрацией, авторизацией и чатом;
- Telegram-бота как альтернативный канал общения;
- Django backend с REST API;
- PostgreSQL как основное хранилище данных;
- локальную языковую модель через Ollama;
- safety-контур, который не доверяет ответам модели без проверки;
- административную панель для управления моделью, справочниками, экстренными ресурсами, контентом и аудитом;
- каталог специалистов и адресов помощи;
- тестовый контур для функциональности, Telegram-бота, каталога, модели, маршрутизации рисков и safety-сценариев.

Ключевая научно-инженерная идея: система не должна быть просто «ботом с нейросетью». В психологически чувствительной области модель обязана работать внутри ограничивающего контура: правила, сценарии скрининга, проверка опасных ответов, аудит маршрутов и понятная эскалация.

## 2. Что важно НЕ переносить из старой курсовой

Старая курсовая использовала упрощенную концепцию:

- центральная роль PHQ-9 и GAD-7;
- Mixtral-8x7B как планируемая модель;
- SBERT/KNN и текстовая база как основной способ генерации рекомендаций;
- старая ER-модель с Session, Recommendation, Questionnaire, Question, Answer;
- идея полноценного дообучения модели на датасете, которого фактически нет.

В дипломе это нужно заменить:

- не делать PHQ-9/GAD-7 ядром работы;
- говорить о стандартизированных опросниках как о расширяемом модуле, а не единственном способе оценки;
- основной акцент перенести на диалоговую QA-систему, локальную LLM, safety-flow и маршрутизацию рисков;
- вместо обещания полноценного fine-tuning описывать реалистичный путь улучшения: prompt-policy, red-team корпус, safety eval, curated dataset, LoRA/fine-tuning как перспективу;
- заменить Mixtral на локально развернутую модель семейства Qwen3 через Ollama.

## 3. Реализованная архитектура

### Frontend

- React 19.
- Vite 7.
- TypeScript.
- React Router.
- Пользовательские блоки: лендинг, описание сервиса, регистрация, вход, чат, каталог специалистов/клиник, пользовательские состояния загрузки и ошибок.
- Чат получает историю сообщений из backend и отправляет новые сообщения через API.

### Backend

- Python 3.12+.
- Django 5.2.
- Django REST Framework.
- PostgreSQL.
- Разделение на приложения:
  - `accounts` — пользователи, роли, привязка каналов;
  - `chat` — чат, сообщения, кризисные события;
  - `neural_engine` — генерация ответов, safety-policy, маршрутизация, аудит;
  - `assessments` — шаблоны опросников, вопросы, прохождения, ответы;
  - `directory` — экстренные ресурсы, специалисты, локации, записи;
  - `platform_ops` — версии модели, админ-контент, модерация;
  - `telegram_bot` — Telegram polling, команды, связка Telegram user -> user_account.

### Neural engine

- Провайдер: Ollama.
- Локальная модель: Qwen3 через Ollama, в текущей конфигурации `qwen3:8b`.
- Системный prompt задает роль русскоязычного помощника психологической поддержки.
- Генерация не является единственным источником решения: ответ проверяется policy layer.
- Ответы модели проходят фильтрацию от опасных формулировок: диагнозы, лекарства, отрицание риска, поощрение самоповреждения, ложное успокоение.

### Telegram bot

- Работает как отдельный процесс, не внутри `runserver`.
- Использует long polling.
- Команды:
  - `/start`;
  - `/help`;
  - `/privacy`;
  - `/emergency`;
- Telegram-пользователь связывается с внутренним `user_account` через `channel_account`.
- История диалога общая по смыслу с веб-чатом: сообщения хранятся в `chat_message`.

## 4. Модель данных

Все ключевые сущности используют UUID.

### Identity & Access

- `user_account` — учетная запись пользователя, email, пароль, имя, статус, staff/superuser.
- `role` — справочник ролей.
- `user_role` — связь пользователя и роли.
- `channel_account` — привязка пользователя к каналу, например web или telegram; хранит external id и журнал сообщений бота для удаления после reset.

### Chat & Safety

- `user_chat` — один активный чат пользователя.
- `chat_message` — полная история сообщений пользователя и бота.
- `crisis_event` — событие риска: уровень, статус, ASQ-screening status, ответы скрининга, действие системы.
- `safety_audit_log` — аудит маршрута: route code, escalation action, human-review flag, model provider, matched rules, policy intervention.
- `emergency_resource` — телефоны и ссылки экстренной помощи.

### Assessments

- `assessment_template` — шаблон опросника.
- `assessment_question` — вопрос внутри шаблона.
- `assessment_session` — прохождение опросника пользователем.
- `assessment_answer` — ответ на вопрос.

Этот блок нужен для расширяемости. Он не должен превращаться в единственный способ диагностики.

### Directory

- `specialist` — психолог или психиатр.
- `specialist_location` — адрес, координаты, цена, город.
- `appointment` — заявка/запись к специалисту.

### Admin & Model

- `neural_model_version` — версия модели: тег, имя, провайдер, safety profile, active flag.
- `moderation_case` — кейс модерации.
- `site_content` — управляемые администратором страницы/FAQ/объявления.

## 5. Safety-flow

Система использует четыре уровня риска:

- `low` — обычный поддерживающий диалог;
- `elevated` — повышенное напряжение, тревога, бессонница, перегрузка; модель отвечает, но добавляется осторожная рекомендация очной помощи при ухудшении;
- `high` — признаки возможного самоповреждения или суицидальной идеации без непосредственного плана; запускается ASQ-сценарий;
- `critical` — непосредственная опасность, намерение, план, метод, подготовка или срочность; ответ модели блокируется, пользователь получает экстренные контакты и рекомендацию немедленно обратиться за помощью.

Логика вдохновлена ASQ Toolkit и C-SSRS-подходом, но система не заявляет клиническую диагностику. В дипломе нужно формулировать аккуратно: «элементы валидированных подходов», «скрининговая логика», «инженерная маршрутизация риска», а не «медицинский диагноз».

## 6. Текущие правила маршрутизации

Система анализирует:

- прямые фразы о нежелании жить;
- идеи самоповреждения;
- упоминание метода;
- намерение;
- срочность;
- подготовку;
- отличие рассказа о себе от рассказа о третьем лице;
- ответы на ASQ-вопросы;
- небезопасные ответы самой модели.

Важный тезис для диплома: модель не имеет права самостоятельно решать кризисный сценарий. Сначала работает safety-router, потом LLM, затем response policy.

## 7. Что можно описывать как перспективу

Это допустимо вставлять в диплом как дальнейшее развитие:

- формирование обезличенного корпуса диалогов с разметкой уровней риска;
- red-team датасет опасных сценариев;
- экспертная разметка психологом/психиатром;
- offline evaluation модели по метрикам safety recall, false negative rate, response usefulness;
- LoRA/fine-tuning только после построения safety-контуров;
- RAG по базе проверенных материалов самопомощи;
- мониторинг качества ответов и human review;
- автоматическое сравнение нескольких моделей: Qwen, Llama, Mistral, DeepSeek;
- интеграция карты специалистов и записи;
- деплой в защищенной среде.

## 8. Что нужно подчеркнуть в дипломе

- Сервис не заменяет врача, психотерапевта, психолога или экстренную помощь.
- Система выполняет предварительную оценку и информационную поддержку.
- Основной вклад работы — архитектура безопасной QA-системы для психологически чувствительного домена.
- Практическая часть реальна: есть backend, frontend, PostgreSQL, Telegram bot, admin, Ollama, тесты.
- Научная часть строится вокруг анализа рисков LLM и разработки safety-flow.
- Полноценное обучение модели — не первый этап; сначала создается контур безопасности и методика оценки.
